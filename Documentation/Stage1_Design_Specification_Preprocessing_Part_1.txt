  Document Purpose & How to Use This Spec

This document is the authoritative design specification for Stage 1 of the Quantum IDS pipeline. It provides a sequential, step-by-step plan that a developer or team can follow to design, validate, and document the data ingestion and taxonomy mapping subsystem before writing a single line of production code.

Each step is structured into four mandatory sections:
• Objective & Sub-Tasks - what must be designed and why
• Key Design Decisions - explicit choices that must be resolved before proceeding
• Risks & Mitigations - failure modes and their countermeasures
• Validation Checkpoints - specific criteria that must be met to exit the step

CRITICAL CONSTRAINT
• Stage 1 produces NO feature transformations. It only assigns master labels, validates data quality, and emits a structured record per network flow.
• All normalization, encoding, scaling, and SMOTE belong exclusively in Stage 2.
• Any design decision that crosses this boundary must be flagged as a scope violation.
  Master Label Taxonomy Reference

The 5-class master taxonomy is the semantic backbone of the entire pipeline. Every subsequent stage - from VAE training loss to SHAP explanations - depends on the accuracy and consistency of these label assignments. The table below is the canonical mapping reference.

AMBIGUOUS MAPPINGS - Require explicit design decision before Step 5
• Fuzzers (UNSW-NB15): Maps to PROBE. Rationale: fuzzing is primarily reconnaissance behaviour even when used as an exploit precursor. Document this decision.
• Generic (UNSW-NB15): Maps to EXPLOIT. Rationale: catch-all for unclassified attack payloads. Flag with mapping_confidence = Low and flag_for_review = true.

• Infiltration (CICIDS-2017): Maps to EXPLOIT. Rationale: multi-stage, but the terminal payload is exploitative. Only ~36 samples - consider exclusion (see Step 5 Decision D5-3).

STEP
1Environment & Repository Setup
Establish the infrastructure before touching any dataset
Objective
Before any dataset is opened, the development environment and project structure must be formally defined. This step ensures reproducibility, enables collaboration, and prevents environment-specific bugs from contaminating experimental results later.

Sub-Tasks
• Define the project's directory structure and document it as the canonical layout for all team members.
• Create a dependency manifest specifying exact library versions for: pandas, numpy, scikit-learn, imbalanced-learn, and any additional ingestion utilities.
• Establish a version control workflow: branching strategy (feature branches per step), commit message convention, and PR review policy.
• Define logging standards: every pipeline stage must emit structured logs with dataset name, step name, record counts before/after, and any anomalies detected.
• Create the canonical output schema document (see Step 7) as a placeholder - it will be filled during this specification process.

KEY DESIGN DECISIONS
• D1-1: Python version (recommend 3.10+ for type hint support across all target libraries).
• D1-2: Virtual environment tool - venv, conda, or poetry. This decision affects how dependency conflicts are resolved when combining quantum and classical libraries later.
• D1-3: Data storage location - local filesystem, cloud bucket (S3/GCS), or shared NFS. This decision affects all downstream file-path references.
• D1-4: Raw data policy - are original dataset files ever modified? Recommend: read-only raw data, all outputs written to separate processed/ directory.RISKS & MITIGATIONS
• Risk: Inconsistent environments across team members cause non-reproducible results. Mitigation: lock all versions in requirements.txt or environment.yml and commit it.
• Risk: Dataset files accidentally modified or overwritten. Mitigation: set raw dataset directories to read-only at OS level and enforce via CI check.
• Risk: Large dataset files (CICIDS-2017 ~8GB) slow down development. Mitigation: create a 10,000-record stratified sample of each dataset for design-phase testing.

STEP
2Dataset Acquisition & Initial Audit
Profile each dataset before designing any transformation
Objective
Each of the three datasets must be obtained, documented, and profiled in its raw, untransformed state. This step produces a Dataset Audit Report - a reference document that all subsequent design decisions in this specification are grounded in measured reality, not assumptions.

Sub-Tasks
• Verify dataset licences and confirm academic/research use is permitted for all three datasets before downloading.
• Document provenance: URL, version, SHA-256 hash, download date, and file format for each dataset. This is required for the paper's reproducibility section.
• For each dataset, compute and document the following profile statistics without any transformation:
? Total record count and class distribution (absolute counts and percentages per native label).
? Feature count and data type per column (numerical, categorical, binary).
? Missing value count and percentage per column.
? Infinity and NaN counts per column - especially critical for CICIDS-2017.
? Min, max, mean, median, and standard deviation for all numerical features.
? Unique value counts for all categorical features.
• Flag NSL-KDD features with near-zero variance (specifically: land, urgent, num_outbound_cmds) and document them.
• Flag UNSW-NB15 raw identity fields (srcip, dstip, sport, dsport) and document the exclusion decision.
• Document the specific columns that contain Infinity values in CICIDS-2017 and their semantic meaning.

KEY DESIGN DECISIONS
• D2-1: How are Infinity values in CICIDS-2017 treated? (Recommended: replace with 99th percentile of that feature - not mean imputation, as Infinity is a computational artifact of zero-duration flows, not a missing value.)
• D2-2: Are near-zero-variance features excluded at ingestion or preserved and flagged? Recommended: preserve but flag, as the VAE may still extract signal from these features across datasets.
• D2-3: Are raw IP/port identity fields from UNSW-NB15 excluded entirely? Recommended: yes, to prevent the model from memorising IP addresses as attack signatures.RISKS & MITIGATIONS
• Risk: Assuming CICIDS-2017 NaN values are missing data and imputing with mean - this is semantically wrong. Mitigation: classify NaN types before imputation (structural vs. missing).
• Risk: Dataset class imbalance is more severe than expected, invalidating the SMOTE strategy in Stage 2. Mitigation: document imbalance ratios now so Stage 2 design is grounded in real counts.
• Risk: Native label vocabulary differs from expected values (typos, case differences in CSVs). Mitigation: document exact unique label strings before building the taxonomy mapping.

STEP
3Per-Dataset Schema Contract Definition
Define the explicit input contract for each dataset ingestion module
Objective
A schema contract is a formal declaration of what a dataset ingestion module expects as input and guarantees as output. By defining these contracts before writing code, the team prevents silent data corruption, enables unit testing, and creates the traceability required for the paper's methodology section.

Sub-Tasks
• Write a schema contract document for each of the three datasets. Each contract must specify:
? Source format: CSV or Parquet, header row present/absent, delimiter, encoding.
? Expected column list: name, data type, and semantic meaning for every column used.
? Columns explicitly excluded: name and documented reason for exclusion (e.g., identity leakage, zero-variance).
? Native label column name and exact expected string values (from Step 2 audit).
? Known data quality issues: which columns carry Infinity, NaN, or negative values that are physically impossible.
• Define what constitutes a 'schema violation' for each dataset - records that fail the contract must be counted, logged, and quarantined (not silently dropped).
• Define the contract version field: each schema contract carries a version string so future dataset updates can be tracked against the original design.

STEP
4Universal Data Quality Gate Design
Define the dataset-agnostic quality filter every record must pass
Objective
The quality gate is a fixed set of validation rules applied identically to all three datasets after schema-aware parsing. It ensures that records entering Stage 2 meet a minimum quality standard without any dataset-specific special-casing. Crucially, it does not transform values - it only accepts, quarantines, or flags records.

Sub-Tasks
• Define the complete set of quality rules. The following are mandatory minimum rules:
? Rule QG-01 (Label Validity): The native label value must exist in the dataset's known label vocabulary from the schema contract. Records with unrecognised labels are quarantined.
? Rule QG-02 (Feature Count): The record must contain exactly the expected number of features after column exclusions. Records with wrong feature count are quarantined.
? Rule QG-03 (Infinity Containment): Any feature containing Infinity must be resolved according to the dataset-specific rule defined in Step 2 (D2-1). This resolution is the ONLY transformation permitted in the quality gate.
? Rule QG-04 (NaN Classification): NaN values must be classified as structural (e.g., single-packet flow statistics) or missing. Structural NaN ? replace with 0. True missing data ? flag record with has_missing = true but do NOT drop.
? Rule QG-05 (Physical Bounds): Certain features have physically impossible negative values (e.g., negative byte counts, negative duration). Records violating physical bounds are flagged.
? Rule QG-06 (Duplicate Detection): Exact duplicate records are detected and logged. One copy is retained; duplicates are counted per dataset in the audit log.
• Define the quarantine policy: quarantined records are written to a separate quarantine file (not deleted) with a quarantine_reason field. Total quarantine rate must be logged.
• Define the quality gate acceptance threshold: if more than 5% of records are quarantined from any single dataset, the pipeline raises a critical alert for human review.
• Design the quality gate output: a boolean pass/fail flag per record, plus a list of rule codes that were triggered (for records that pass with warnings).

KEY DESIGN DECISIONS
• D4-1: What is the 99th percentile replacement value for CICIDS-2017 Infinity? It must be computed on the TRAINING split only, then applied to both train and test. The exact value must be stored as a pipeline artefact.
• D4-2: How are structural NaN values distinguished from true missing data? Design decision: define structural NaN as NaN in a statistical feature (std, min, max, mean) when the packet count is 1 - this is the known CICFlowMeter degenerate case.
• D4-3: Should records with bounds_violation flags be passed to Stage 2 with the flag, or corrected? Recommended: pass with flag - Stage 2 owns the correction decision as part of normalization.RISKS & MITIGATIONS
• Risk: The 5% quarantine threshold is exceeded for CICIDS-2017 due to Infinity prevalence - triggering a false critical alert. Mitigation: compute expected quarantine rate from the audit in Step 2 and set the threshold accordingly.
• Risk: QG-03 replacement value is computed on the full dataset (train+test) causing data leakage. Mitigation: explicitly document that the 99th percentile is computed on training data only and stored as a serialised artefact.
• Risk: QG-06 duplicate detection is computationally expensive on CICIDS-2017 (8GB). Mitigation: use hashing-based deduplication on a subset of key columns rather than full row comparison.

STEP
5Master Label Taxonomy Mapping Specification
Define every label mapping with confidence rating and ambiguity resolution
Objective
This step produces the official, versioned taxonomy mapping specification - a document that determines exactly how every native label in every dataset is converted to a master label. This document is a primary research artefact and will be referenced directly in the paper's methodology section. Ambiguous mappings must be explicitly resolved here, not deferred to implementation.

Sub-Tasks
• Using the native label vocabulary from Step 2 (not documentation - actual observed string values), define the complete mapping table for all three datasets.
• Assign a mapping confidence rating to each mapping: High, Medium, or Low. Criteria:
? High: the native label has a clear, unambiguous semantic match to one master label.
? Medium: the native label partially matches multiple master labels but has a clear dominant mapping.
? Low: the native label is a catch-all or multi-stage category with no dominant mapping.
• For every Medium and Low confidence mapping, write an explicit justification paragraph (3-5 sentences) documenting the rationale. These justifications go directly into the paper.
• Resolve Decision D5-3: whether to include or exclude the CICIDS-2017 Infiltration class (~36 samples). Document the decision and its rationale.
• Resolve Decision D5-4: whether the UNSW-NB15 Generic class is mapped to EXPLOIT or excluded due to semantic ambiguity.
• Define the cross-dataset class distribution table: after mapping, how many samples belong to each master label from each dataset? This reveals whether MALWARE is statistically viable.

Feature Array Alignment Policy
The 196-element feature array is the most consequential design decision in Stage 1. The policy below governs how datasets with fewer than 196 native features populate this array.

FEATURE ALIGNMENT POLICY - Version 1.0
• The 196-element feature array is organised into 8 semantic groups (Flow Identity, Protocol State, Service Context, Traffic Rate, Statistical Profile, Connection Behaviour, Content Features, Engineered Features).
• Each semantic group reserves a fixed index range within the 196-element array. These ranges are defined in the Semantic Feature Schema document (produced in Stage 2 design).
• When a dataset has no feature for a group's index range, that range is populated with a sentinel value of -1.0 (not 0.0, which is a valid normalised value). Stage 2 must treat -1.0 as 'feature absent' and handle it distinctly from true zero values.
• The sentinel value approach is preferred over zero-padding because it preserves the semantic distinction between 'this feature is zero' and 'this feature does not exist in this dataset'. The VAE encoder must learn to ignore -1.0 sentinel values, not conflate them with zero-rate traffic.
• The mapping from dataset-native column to semantic group index is defined in a second YAML artefact: feature_alignment_map.yaml. This file is versioned and committed to the repository.
KEY DESIGN DECISIONS
• D7-1: Sentinel value of -1.0 vs. learned default per semantic group. Recommended: -1.0 is simpler and forces the VAE to learn a 'missing feature' representation. A learned default adds complexity with marginal benefit at this stage.
• D7-2: Should the output record be stored as Parquet or CSV? Recommended: Parquet - it preserves data types (critically, boolean fields and float precision), is significantly faster to read, and is natively supported by all ML frameworks.
• D7-3: Should the feature array be stored as a single column of arrays or as 196 individual columns? Recommended: 196 individual columns for Parquet storage; the array is assembled in memory at training time by the Stage 2 loader.RISKS & MITIGATIONS
• Risk: The sentinel value -1.0 falls outside the Min-Max [0,1] range, causing Stage 2's normaliser to produce unexpected outputs. Mitigation: Stage 2 must apply a sentinel mask before normalization - document this as a Stage 2 requirement in this specification.
• Risk: schema_version mismatches between Stage 1 and Stage 2 cause silent incompatibilities as the project evolves. Mitigation: Stage 2 must validate schema_version on first record read and raise if it doesn't match the expected version.
• Risk: The feature alignment YAML becomes a maintenance burden as datasets are updated. Mitigation: freeze the YAML for v1.0; any dataset update triggers a formal schema review, not an ad-hoc file edit.

Objective
This final step treats the design specification itself as a deliverable that requires the same rigour as code.
