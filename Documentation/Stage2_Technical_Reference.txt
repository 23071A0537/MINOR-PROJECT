1. Executive Summary

Stage 2 is the classical preprocessing stage of the Explainable Hybrid Quantum-Classical Network Intrusion Detection System (NIDS). It bridges Stage 1 (raw data ingestion and taxonomy mapping) with Stage 3 (Variational Autoencoder compression) by transforming three structurally different network traffic datasets into a single, clean, unified, numerically normalised feature matrix suitable for deep learning and quantum computation.

The pipeline executes eleven ordered steps, each with a strict data-leakage-prevention requirement: no statistical parameter - encoder frequency, scaler minimum, scaler maximum, imputation median - is ever computed on test data or on the full pre-split corpus (with the sole documented exception of two infinity replacement values carried forward from Stage 1, which are explicitly acknowledged and traceable).

3. Pipeline Architecture Overview

The eleven pipeline steps are ordered to guarantee strict leakage prevention. The critical constraint is that no fitting operation - encoding, imputation, or scaling - can observe test data.

Critical Ordering Constraint
Steps 1-3 must complete before any fitting operation begins. The train/test split (Step 3) acts as a firewall - everything from Step 4 onwards fits only on training data.
Violating this ordering constitutes data leakage and invalidates all downstream evaluation metrics.

Step 2 - Feature Alignment
Purpose
The three datasets have different feature sets. NSL-KDD captures network connection metadata (36 features), UNSW-NB15 captures packet-level statistics (42 features), and CICIDS-2017 captures flow-level CICFlowMeter statistics (77 features). Stage 2 must unify these into a single feature matrix where every record has exactly the same columns.

The alignment strategy uses sentinel values to represent 'this feature was not measured for this dataset' - a fundamentally different signal from 'this feature has a value of zero'. Using different sentinel values by type (numeric vs categorical) prevents confusion during encoding.

Sentinel Policy (D7-1)
Feature TypeSentinel ValuePost-scaling ValueRationaleNumeric-1.00.0Distinguishable from valid values which are always = 0 in network traffic. Zeroed after scaling.Categorical (string)'__ABSENT__'0.0 (all-zeros OHE or freq=0)String sentinel prevents confusion with real category values. Maps to zero vector after encoding.
Absent Feature Counts
DatasetOwn FeaturesAbsent FeaturesSentinel Rate per RecordNSL-KDD36 of 15411876.6% of positions are sentinelUNSW-NB1542 of 15411272.7% of positions are sentinelCICIDS-201777 of 1547750.0% of positions are sentinel
Why the high sentinel rates are acceptable
The sentinel mask (Step 5) explicitly tracks which positions are absent. The VAE reconstruction loss in Stage 3 is masked so absent feature positions contribute zero weight.
Post-scaling, absent positions are exactly 0.0 - the model receives a clean signal and the sentinel mask provides the auxiliary information needed to distinguish zero-valued real features from absent ones.
Implementation Note: Column Order
After padding, all three DataFrames are forced to the same sorted column order before concatenation. This deterministic ordering is preserved through the entire Stage 2 pipeline and written to stage2_feature_names.json. Every downstream stage (Stage 3 VAE, Stage 5 Ensemble) must read this file rather than assuming a column order.

Step 3 - Dual-Stratified Train / Test Split
Purpose
The train/test split must occur before any encoder or scaler is fitted. This is the most important leakage-prevention step in the pipeline.

Stratification Strategy
A simple stratification on master_label alone is insufficient because the three datasets have very different class distributions. For example, CICIDS-2017 has no MALWARE records from NSL-KDD, and UNSW-NB15 has a much larger PROBE fraction than CICIDS-2017. If the test set happened to draw disproportionately from one dataset, the evaluation metrics would not reflect cross-dataset generalisation.

The solution is a compound stratification key: master_label concatenated with source_dataset (e.g. 'MALWARE__UNSW-NB15', 'DoSD__CICIDS-2017'). This creates 15 strata and ensures that both the class distribution and the dataset provenance are proportionally preserved in both splits.

Step 4 - Categorical Encoding
Purpose
The three datasets contain several categorical (string) columns that must be converted to numeric values before scaling and model training. Two encoding strategies are applied depending on column cardinality, with all encoders fitted exclusively on the training split.

Cardinality-Based Strategy
A threshold of 10 unique real (non-sentinel) values determines the encoding method. Columns at or below the threshold use One-Hot Encoding (OHE); columns above use Frequency Encoding.

ColumnDataset(s)Real Unique ValuesMethodRationaleprotocol_typeNSL-KDD3 (tcp, udp, icmp)OHELow-cardinality, interpretableflagNSL-KDD11 TCP flag statesFrequencyModerate cardinality; OHE would add 11 columns for a single datasetserviceNSL-KDD + UNSW-NB15~80 combinedFrequencyHigh cardinality; OHE would add ~80 sparse columnsprotoUNSW-NB15~130FrequencyVery high cardinality; OHE impracticalstateUNSW-NB15~10OHE or Frequency**Determined at runtime based on actual cardinality
FrequencyEncoder Design
The custom FrequencyEncoder maps each category value to its relative frequency in the training set. This preserves ordinal information (common categories get higher values) without the dimensionality explosion of OHE, and handles unseen test values gracefully.

OHE Design
sklearn's OneHotEncoder is fitted on non-sentinel training values. The handle_unknown='ignore' parameter ensures unseen test values produce an all-zeros row. Sentinel (__ABSENT__) rows are forced to all-zeros independently of the OHE output, and the feature lineage dictionary maps every OHE-derived column back to its parent so the sentinel mask can be propagated correctly in Step 5.

Post-Encoding Column Count
The exact post-encoding column count depends on the runtime cardinality of categorical columns in the training split. It is always = 154 (the pre-encoding union). The authoritative count is written to stage2_feature_names.json and must be read by Stage 3 to configure the VAE input layer. It must not be hardcoded in any downstream stage.

Step 5 - Sentinel Mask Creation
Purpose
The sentinel mask is a boolean array of shape (N × F) where N is the number of records and F is the number of post-encoding features. A value of True at position [i, j] means that feature j was absent for record i because record i comes from a source dataset that does not include that feature.

Critical: Must be created BEFORE scaling (D7-1)
If the sentinel mask were created after scaling, the MinMaxScaler would have already used the -1.0 sentinel values in its min/max computation, distorting the scaling of real values.
The mask is built from metadata (source_dataset column) and the feature lineage dictionary - no feature values are inspected.
Construction Algorithm
6. For each dataset (NSL-KDD, UNSW-NB15, CICIDS-2017), compute the set of post-encoding columns whose pre-encoding parent was NOT present in that dataset's original feature set.
7. For all records from that dataset, set sentinel_mask[i, j] = True for those column indices.
8. Records not from any dataset default to False (no sentinel positions - this should never occur in practice).

Usage Across the Pipeline
StageHow Sentinel Mask is UsedEffectStep 9 (Scaling)Exclude sentinel positions from min/max computationScaler statistics reflect only real feature valuesStep 9 (Post-scale)Zero out sentinel positions after scalingAbsent features become 0.0; real features in [0.0, 1.0]Stage 3 (VAE)Mask the per-feature reconstruction lossVAE does not penalise reconstruction of absent featuresStage 5 (Ensemble)Passed to XGBoost and RF as auxiliary feature or ignoredTree models handle 0.0 absent naturally; mask available for analysis
Step 6 - CICFlowMeter Integer-Overflow Column Clipping
Background
CICFlowMeter, the tool used to generate CICIDS-2017, has a known integer overflow bug affecting several columns. When a computed value overflows a signed 32-bit integer, it wraps to a large negative number. For example, the Fwd Header Length column has a minimum value of approximately -3.22 × 10¹° in the raw data, which is physically impossible for a TCP header length field.

Stage 1 flagged these records with bounds_violation=True (50.7% of all CICIDS-2017 records) but did not modify the values, following the Stage 1 principle of raw feature preservation. Stage 2 is responsible for correcting these values before scaling.

Strategy: Clip, Do Not Exclude
Records are NOT excluded. The overflow is an instrument artefact, not a data collection error. The affected records are legitimate network flows - only specific computed columns contain invalid values. Excluding 50.7% of CICIDS-2017 (over 1.27 million records) would severely damage the representativeness of the training corpus.

Note: Clipping applied to CICIDS-2017 rows only
The clipping logic identifies rows by source_dataset metadata and applies corrections only to CICIDS-2017 records. NSL-KDD and UNSW-NB15 records are not touched.
The clip bounds are domain-determined (physical constraints of TCP/IP), not learned from data, so there is no leakage concern from applying the same bounds to both train and test.
Step 7 - Imputation for True Missing Values
Context
After feature alignment and sentinel padding, the only remaining NaN values are 1,358 CICIDS-2017 records where the Flow Bytes/s column is genuinely missing (not an overflow, not a sentinel). These are records where CICFlowMeter failed to compute a byte rate, typically for zero-duration flows.

These records are identified in Stage 1 metadata via has_missing=True. Stage 2 Forward Requirement #5 from Stage 1 specifies that these records must not have SMOTE applied without prior imputation.

Imputation Design
PropertyValueStrategyMedian imputation (SimpleImputer, strategy='median')Fit onTraining set only - prevents test-set median from leaking into imputed valuesApplied toTrain and test (transform only for test)Affected records~1,086 train (1,358 × 0.80) + ~272 test (1,358 × 0.20)Post-imputation NaN checkHard assert: np.isnan(X).any() must be False for both train and test
Median is chosen over mean because Flow Bytes/s has a highly right-skewed distribution (large outliers from high-throughput flows). The median is a more robust central tendency estimate and will not be distorted by the extreme values observed in DDoS attack flows.

Step 8 - QG-03 Leakage Documentation
Background
In Stage 1, the Infinity containment gate (QG-03) replaced infinite values in Flow Bytes/s and Flow Packets/s with 99th percentile values computed on the full CICIDS-2017 dataset. Since Stage 1 does not own the train/test split, these replacement values necessarily include information from what will become the test set.

Acknowledged Data Leakage Vector
The QG-03 replacement values for Flow Bytes/s (? 12,333,333) and Flow Packets/s (? 2,000,000) were computed on the full CICIDS-2017 dataset (2,830,743 records including Wednesday traffic).
This constitutes a minor data leakage: the test set Infinity values were replaced using a statistic that includes test-set records.
The leakage is quantified as affecting 1,509 + 2,867 = 4,376 values out of 2,830,743 × 77 = 217,967,011 total CICIDS-2017 values (0.002%).
The structural fix (apply_quality_gate() accepts inf_replacement_values parameter) means this can be eliminated entirely by computing train-only p99 after Stage 2 split and re-running Stage 1. This is recommended for the camera-ready paper version.
Step 8 Operations
• Load qg03_replacement_values.json and log the leakage acknowledgement.
• Verify that no Infinity values remain in either the train or test split (assert np.isinf(X).any() == False).
• If Infinity is found: halt execution with a descriptive error - this would indicate the Stage 1 QG-03 was not applied or the data was loaded incorrectly.

Step 9 - Sentinel-Aware MinMax Scaling
Purpose
All feature values must be normalised to [0.0, 1.0] before being fed to the Variational Autoencoder (Stage 3). The VAE bottleneck output is subsequently scaled to [0, p] for quantum angle encoding, so the model is sensitive to the absolute scale of input values.

Why Standard MinMaxScaler Fails Here
The standard sklearn MinMaxScaler computes per-column minimum and maximum over all values. If sentinel values (-1.0) are included, the per-column minimum becomes -1.0 for every column that has any sentinel rows, causing the scaler to stretch the entire feature range from [-1, max_real] to [0, 1]. This would compress all real values into a sub-range of [0, 1] and would make the scaled outputs incomparable across datasets.

SentinelAwareMinMaxScaler Design
class SentinelAwareMinMaxScaler:
    def fit(self, X, sentinel_mask):
        for j in range(X.shape[1]):
            real_vals = X[~sentinel_mask[:, j], j]  # exclude sentinel rows
            real_vals = real_vals[np.isfinite(real_vals)]
            if len(real_vals) > 0:
                self.col_min_[j] = real_vals.min()
                self.col_max_[j] = real_vals.max()
            # else: all-sentinel column ? identity (min=0, max=1)
 
    def transform(self, X):
        scale = np.where(col_max > col_min, col_max - col_min, 1.0)
        X_scaled = (X - col_min) / scale
        return np.clip(X_scaled, 0.0, 1.0).astype(np.float32)
 
# After transform: zero out sentinel positions
X_scaled[sentinel_mask] = 0.0
Post-Scaling State
Position TypeScaled ValueMeaningReal feature, present in dataset[0.0, 1.0]Normalised feature value; 0.0 = minimum, 1.0 = maximum seen in trainingAbsent feature (sentinel)0.0 (forced)Feature not measured for this dataset; sentinel mask = TrueReal feature value = 00.0 (natural)Feature present and genuinely zero; sentinel mask = False
Note on 0.0 Ambiguity
Both 'feature is absent' and 'feature value is zero' produce 0.0 after scaling. The sentinel mask is the only way to distinguish them.
Stage 3 (VAE) receives both the scaled feature array and the sentinel mask. The reconstruction loss is weighted by (1 - sentinel_mask) so the VAE only learns from real feature values.
Step 10 - SMOTE Oversampling
Purpose
Two classes are significantly underrepresented in the training set after the generic records exclusion: EXPLOIT (~47,866 samples, 2.04%) and MALWARE (~4,774 samples, 0.20%). Without oversampling, the classifier will be strongly biased towards the majority class (NORMALL at 77.41%) and will have poor sensitivity for the threat classes that matter most operationally.

SMOTE vs Other Techniques
Synthetic Minority Oversampling Technique (SMOTE) was chosen over simple duplication because duplicated samples do not add information - they merely increase the loss weight of existing samples without helping the model generalise. SMOTE generates interpolated synthetic samples between existing minority-class records in feature space, improving decision boundary learning for minority classes.

ADASYN (Adaptive Synthetic Sampling) was considered but rejected because it focuses synthesis on hard-to-classify boundary samples, which would risk generating synthetic MALWARE samples close to the NORMALL decision boundary - potentially teaching the model incorrect patterns for the most security-sensitive class.

Application Order
SMOTE is applied AFTER scaling. This is intentional: synthetic sample generation works in the normalised [0, 1] feature space, which means nearest-neighbour distance calculations are computed on comparable scales across all features. Applying SMOTE before scaling would cause features with large raw ranges (e.g. src_bytes in NSL-KDD, up to 1.38 × 10?) to dominate nearest-neighbour calculations.

Sentinel Mask for Synthetic Samples
SMOTE-generated synthetic samples do not come from any real source dataset, so they have no inherent sentinel mask. Two observations motivate the assignment strategy used:
• All MALWARE samples originate exclusively from UNSW-NB15 (Backdoors, Shellcode, Worms). All synthetic MALWARE samples receive the UNSW-NB15 sentinel pattern (112 absent positions, 42 real positions).
• EXPLOIT samples are primarily from UNSW-NB15 (~103,396 records) with a smaller contribution from NSL-KDD (~4,001 records) and CICIDS-2017 (~11,306 records). Synthetic EXPLOIT samples receive the UNSW-NB15 sentinel pattern as the majority-source pattern.

This assignment is an approximation. Because absent feature positions are already 0.0 after scaling, and SMOTE interpolation between two same-dataset records naturally produces 0.0 for absent positions, the practical impact of this approximation is minimal. The Stage 3 VAE forward requirement notes that training the VAE only on real (non-synthetic) samples is an option worth evaluating.

6. Key Design Decisions and Rationale

DD-1: Frequency Encoding over Target Encoding for High-Cardinality Columns
Target encoding (replacing a category with its mean label value) was considered for the service and proto columns. It was rejected because target encoding is highly sensitive to class imbalance - with NORMALL at 77% of training data, the mean label value for most categories would be close to 0 (NORMALL) regardless of the category's actual threat relevance. This would cause the encoder to assign nearly identical values to 'http' (predominantly normal traffic) and obscure attack-bearing service categories.

Frequency encoding uses only the marginal distribution of each category (how often it appears), not its conditional distribution with respect to the label. This makes it immune to class imbalance effects and does not constitute supervised learning that could leak label information.

DD-2: Sentinel Mask as an Auxiliary Signal rather than Feature Deletion
An alternative design would delete cross-dataset-absent features entirely and train three separate models - one per dataset. This was rejected because the goal of Stage 3 is to learn a unified 8-dimensional representation that captures network intrusion patterns across all three datasets. A shared representation enables the quantum classifier to generalise across data sources.

The sentinel mask approach retains all features in a single unified model. The VAE learns to reconstruct real features while ignoring absent positions (via loss masking), effectively learning dataset-conditioned representations within a single model.

DD-3: MinMax Scaling over StandardScaler (Z-score)
StandardScaler was considered because it makes no assumption about the distribution shape. It was rejected because network traffic features are heavily right-skewed (many near-zero values, occasional extreme outliers in byte counts and packet rates). Z-score normalisation of such features would produce a wide range of positive values for outliers without a guaranteed bound.

MinMax scaling to [0, 1] provides a hard bound that is critical for the downstream quantum angle encoding in Stage 4. The VAE bottleneck output must be bounded to [0, p] for valid qubit rotation angles, and this is easier to guarantee when the VAE input is bounded to [0, 1].

DD-4: SMOTE in Feature Space After Scaling
Some implementations apply SMOTE before scaling to avoid scale effects on the KNN distance metric used internally by SMOTE. However, with raw network traffic features ranging over many orders of magnitude (src_bytes: 0 to 1.38 × 10?; rate: 0 to 106), unscaled KNN distances would be entirely dominated by the largest-magnitude features. Applying SMOTE after scaling ensures each feature contributes proportionally to nearest-neighbour calculations.

DD-5: Generic Records Excluded from EXPLOIT Class
UNSW-NB15 contains 58,871 records labelled 'Generic' - a dataset-specific category with no agreed real-world threat analogue. Earlier versions of the pipeline mapped these to EXPLOIT with Low confidence and a flag_for_review flag. Following team discussion, these records were excluded in Stage 1 (version 1.1).

The rationale: including Generic records would mean approximately half of the EXPLOIT training samples came from a category whose threat semantics are undefined. SHAP and LIME explanations generated for EXPLOIT predictions would reflect patterns from Generic traffic, making it impossible to claim that the model has learned 'what exploitation attempts look like' in any operationally meaningful sense. Exclusion reduces EXPLOIT training samples by ~55% but results in a cleaner, more defensible class definition.
