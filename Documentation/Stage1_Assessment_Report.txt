  Executive Summary

The notebook Stage1_Preprocessing_Complete.ipynb is not a completed Stage 1 implementation. While it demonstrates sound intent and correctly implements several isolated components - particularly the quality gate logic and the NSL-KDD schema contract - it contains six critical findings that make the current output unsuitable as input for Stage 2 or Stage 3.

The most consequential problem is an architectural one: the notebook combines Stage 1, Stage 2, and a portion of Stage 3 into a single pipeline, then produces the wrong output. It outputs an 8-dimensional PCA-reduced array, which directly conflicts with the VAE-based dimensionality reduction strategy that is the primary innovation of the research project. The VAE in Stage 3 cannot be trained if PCA has already collapsed the feature space. Additionally, the scaler and imputer are fitted on the full dataset before the train/test split, introducing data leakage into every experiment conducted with this notebook.

  VERDICT    Stage 1 is NOT Complete
• The notebook must be substantially restructured before it can be considered a correct Stage 1 implementation.
• The PCA cell (Step 15) must be removed entirely - this conflicts with the project's core VAE architecture.
• Data leakage must be corrected: scaler and imputer must be fitted on training data only, after the split.
• The output record schema (11 mandatory fields) is not produced - original_label, source_dataset, mapping_confidence, and flag_for_review are all absent.
• Only one CICIDS-2017 daily file is loaded - the dataset is severely under-represented.
• Stage 2 transformations (OHE, imputation, scaling, SMOTE) belong in a separate Stage 2 notebook.

  Section 2 - Critical Findings

Critical findings are blockers. The notebook cannot be used for downstream stages until every critical finding below is resolved. Each finding includes the cell reference, the exact problem, and the required correction.

  CF-01    PCA Is an Architecture Violation - Must Be Removed Entirely
• Location: Cell 44 (Step 15 - PCA Dimensionality Reduction to 8 Features).
• The project's primary innovation is using a Variational Autoencoder (Stage 3) for probabilistic dimensionality reduction. The design specification explicitly states: 'Feature reduction is NOT required at this stage because I will apply a VAE later.'
• By applying PCA before the VAE exists, the notebook produces an 8-dimensional output instead of the full-width feature matrix that Stage 2 and Stage 3 require. There is nothing for the VAE to learn from - the information has already been collapsed.
• PCA is also a deterministic linear transform that cannot capture the non-linear, probabilistic structure that makes the VAE useful for separating attack distributions in latent space.
• Required fix: Delete Cell 44 entirely. The notebook's output should be the full-width normalised feature matrix, not an 8-dimensional PCA projection. The QUANTUM_DIM=8 constraint belongs to the VQC circuit in Stage 4, reached via the VAE bottleneck in Stage 3.
  CF-02    Data Leakage - Scaler and Imputer Fitted Before Train/Test Split
• Location: Cell 36 (imputer.fit_transform(X)) and Cell 38 (scaler.fit_transform(X_arr)), both executed BEFORE Cell 40 (train_test_split).
• fit_transform() both learns statistics from and transforms the data simultaneously. When applied to the full dataset, the scaler learns the min/max and the imputer learns the median including test set values. Every model trained downstream has implicitly seen test data statistics.
• This invalidates all accuracy and F1 metrics produced by this pipeline. Results will be optimistically inflated and non-reproducible when the pipeline is applied to genuinely unseen data.
• Required fix: The train/test split must be moved to immediately after taxonomy mapping (before OHE, imputation, and scaling). Then fit the imputer on X_train only (.fit()), transform both X_train and X_test (.transform()). Same pattern for the scaler.
• This also applies to QG-03: the 99th percentile replacement value is computed on the full dataset in apply_quality_gate(). That function must receive only training data, then its replacement values must be stored as an artefact and applied separately to test data.
  CF-03    Output Record Schema Not Produced - 7 of 11 Mandatory Fields Missing
• The Stage 1 Design Specification v1.0 defines an 11-field canonical output record. The notebook produces none of the metadata fields.
• MISSING: source_dataset - there is no way to identify which dataset a record came from in the output arrays.
• MISSING: original_label - the native label string is discarded after mapping; traceability to the source is lost.
• MISSING: mapping_confidence - not computed or attached anywhere. This field drives the SMOTE strategy in Stage 2 and the ablation study in the paper.
• MISSING: flag_for_review - not implemented. Low-confidence mappings (Generic, Infiltration) are silently included with no marker.
• MISSING: ingestion_timestamp - not attached to records.
• MISSING: schema_version - defined as a constant but never attached to the output records.
• MISSING: quality_flags list - the quality gate computes this correctly but Cell 28 immediately drops quality_pass, has_missing, and bounds_violation before they can be used.
• Required fix: Refactor the pipeline to emit a structured DataFrame/Parquet file with all 11 output schema fields as columns, not raw numpy arrays.
  CF-04    CICIDS-2017 Is Loaded from One Daily File Only - Dataset Severely Under-Represented
• Location: Cell 10 - only 'Friday-WorkingHours-Afternoon-DDos.pcap_ISCX.csv' is loaded.
• The design specification states: 'multiple daily files must be concatenated.' CICIDS-2017 has eight daily files covering different attack types across different days (Monday through Friday, multiple sessions).
• Loading only the Friday DDoS afternoon file means the pipeline has no PortScan traffic, no Brute Force traffic, no Web Attacks, no Bot traffic, and no Heartbleed samples from CICIDS-2017. The PROBE and EXPLOIT master labels will be systematically under-represented.
• The audit report, quality gate statistics, and taxonomy distribution shown in the notebook are therefore not representative of the actual CICIDS-2017 dataset.
• Required fix: Load and concatenate all daily CICIDS-2017 CSV files. After concatenation, verify column name consistency across files (some files have slight differences) and document the intersection of common columns per Decision D3-2.
  CF-05    Stages 1 and 2 Are Conflated - Pipeline Scope Boundary Violated
• Stage 1 (this notebook) should output raw, quality-gated, labelled records with no feature transformation beyond Infinity/structural NaN resolution (as permitted by QG-03 and QG-04).
• The following operations belong exclusively in Stage 2 and must not appear in Stage 1:
• Cell 34 - One-Hot Encoding: categorical feature transformation is Stage 2.
• Cell 36 - SimpleImputer (median): remaining missing value imputation is Stage 2.
• Cell 38 - MinMaxScaler: feature normalisation to [0,1] is Stage 2.
• Cell 40 - train_test_split: the split should occur at the Stage 2 boundary after all three datasets are combined and the 196-feature aligned corpus is assembled.
• Cell 42 - SMOTE: class balancing is Stage 2.
• Cell 44 - PCA: this is neither Stage 1 nor Stage 2 - it must be replaced by the VAE in Stage 3.
• Required fix: Create a Stage 2 notebook for all operations listed above. Stage 1 should terminate after Cell 32 (taxonomy mapping), saving the structured output record as a Parquet file.
  CF-06    No Cross-Dataset Unified 196-Feature Corpus Is Produced
• The pipeline processes one dataset at a time via the DATASET variable switch. This produces three independent single-dataset pipelines, not a unified cross-dataset corpus.
• The design specification's central structural requirement is that all three datasets are combined into a single 196-dimensional feature space using semantic feature group alignment and the -1.0 sentinel value policy for absent features.
• A model trained on nsl_kdd alone cannot generalise across datasets. The quantum advantage claim requires evaluation on all three datasets jointly or with explicit cross-dataset transfer.
• Required fix: After Stage 1 processes all three datasets independently, a concatenation step must assemble the unified corpus. The 196-feature alignment map (feature_alignment_map.yaml) must be implemented, and the -1.0 sentinel policy must be applied for absent feature groups.

  Section 3 - Major Findings

Major findings are not immediate blockers but will cause incorrect results, invalid paper claims, or significant technical debt if not resolved before Stage 2 implementation begins.

MF-01 - Generic (UNSW-NB15) Mapped to DoS, Not EXPLOIT (Cell 32)
Specification decision D5-4: 'Generic' should map to EXPLOIT (class 3) with Low confidence. The notebook maps it to class 1 (DoS) with a comment saying 'DoS per guide' - this contradicts the specification and the semantic rationale. Generic attacks use exploit payloads, not volume-based denial-of-service techniques.
Required fix: Change UNSW_MAP['generic'] from 1 to 3. Add mapping_confidence = 'Low' and flag_for_review = True to records mapped from Generic.

MF-02 - Unknown Label Fallbacks Are Silent and Incorrect (Cell 32)
NSL-KDD unknown labels fall through to return 1 (DoS). UNSW-NB15 unknown labels become 1 via .fillna(1). CICIDS-2017 unknown labels become 0 (NORMALL) via .fillna(0). Silently assigning unknown attack traffic as NORMALL (class 0) is particularly dangerous - it will train the model to classify that attack pattern as legitimate traffic. Unknown labels should either be quarantined via QG-01 or explicitly handled with a documented decision.
Required fix: Add a post-mapping assertion that counts unmapped labels. Log them and quarantine them rather than assigning a default class.

MF-03 - service Column (NSL-KDD) One-Hot Encoded Despite High Cardinality Warning (Cell 34)
The design specification flagged that NSL-KDD's service column has 70 unique values, and recommended target encoding or frequency encoding instead of OHE. Using pd.get_dummies() on service generates 70 additional binary columns from a single feature. This inflates the feature space disproportionately and makes the resulting matrix dataset-specific - UNSW-NB15 and CICIDS-2017 will have entirely different OHE column structures, making cross-dataset alignment impossible.
Required fix: Replace OHE for service (NSL-KDD) and proto (UNSW-NB15, ~130 unique values) with frequency encoding or target encoding in Stage 2.

MF-04 - Near-Zero Variance Features Not Excluded for NSL-KDD
The schema contract (Cell 19) correctly identifies land, urgent, and num_outbound_cmds as near-zero variance features. However, the DROP_COLS list in Cell 28 only drops difficulty_level. The three flagged near-zero variance features are retained in the feature matrix, adding noise dimensions to every NSL-KDD experiment.
Required fix: Add 'land', 'urgent', 'num_outbound_cmds' to the NSL-KDD entry of DROP_COLS.

MF-05 - Quality Metadata Columns Dropped Before They Can Be Used (Cell 28)
The quality gate correctly computes quality_pass, has_missing, and bounds_violation and attaches them to df_clean. Cell 28 then immediately drops them via the QUALITY_META list before any downstream component has a chance to use them. The has_missing flag is particularly important: the spec states 'Stage 2 must not apply SMOTE to records where has_missing = True without imputation.' If the flag is dropped, Stage 2 cannot respect this requirement.
Required fix: Remove QUALITY_META from the drop logic. Preserve these fields in the Stage 1 output record, and handle them appropriately in Stage 2.

MF-06 - Dual Stratification Not Implemented in Train/Test Split (Cell 40)
The specification requires stratifying on both master_label AND source_dataset simultaneously to prevent any single dataset from dominating either the train or test split. The current implementation only stratifies on y_arr (master_label alone). Since the pipeline currently processes one dataset at a time, this is partially masked, but it will become a significant issue once the unified cross-dataset corpus is assembled.
Required fix: Create a combined stratification key (e.g., source_dataset + '_' + str(master_label)) and use it as the stratify argument.

MF-07 - QG-03 Replacement Values Computed on Full Dataset (Data Leakage Variant)
Inside apply_quality_gate(), the 99th percentile for Infinity replacement is computed from all finite values in the column passed to the function. In Cells 22-24, the functions are called on the full uncleaned DataFrames (not on the training split). This means test-set flow statistics influence the Infinity replacement value - a subtler form of the same leakage identified in CF-02.
Required fix: Apply QG-03 Infinity handling in two phases: (1) compute replacement values on training data only and serialise them, (2) apply those stored values to test data without recomputing.

MF-08 - SMOTE Applied Before Addressing has_missing Records (Cell 42)
SMOTE interpolates between minority class neighbours to generate synthetic samples. If records with has_missing = True are present in the training set (because the flag was dropped in Cell 28), SMOTE will interpolate across NaN values, producing synthetic samples with NaN features. This is a silent data corruption issue that will cause unpredictable downstream model behaviour.
Required fix: Filter out has_missing records (or impute them) before applying SMOTE. This is only possible if the has_missing flag is preserved (see MF-05).
