1. Project Overview & Pipeline Architecture

This document provides a complete technical explanation of the first two stages of the Explainable Hybrid Quantum-Classical Network Intrusion Detection System (NIDS). The system is designed to classify network traffic flows into five threat categories using a pipeline that combines classical machine learning preprocessing, neural compression via a Variational Autoencoder (VAE), and a Variational Quantum Circuit (VQC) classifier, backed by SHAP and LIME explainability.

Why a six-stage pipeline?  Network intrusion detection on real-world multi-source data requires rigorous preprocessing before any model training can begin. Raw network traffic datasets differ in feature sets, label schemes, quality, and scale. Each stage of this pipeline solves one class of problem in isolation, making every step auditable and reproducible.

StageNameInputOutput / Purpose1Data Ingestion & Taxonomy3 raw CSV datasets3 Parquet files, unified 5-class labels, 154 union features2Classical PreprocessingStage 1 Parquet filesEncoded, scaled, SMOTE-balanced arrays (2,367,588 × 167)3VAE CompressionStage 2 arrays8D latent representations bounded to [0, p]4Quantum Circuit (VQC)8D VAE bottleneckQuantum probability vector via 8-qubit VQC5Hybrid EnsembleVQC + classical featuresCombined VQC + XGBoost + RandomForest predictions6ExplainabilityEnsemble + featuresSHAP global importance + LIME per-prediction explanations

2. Stage 1 - Data Ingestion & Taxonomy Mapping

Stage 1 is the data foundation of the entire pipeline. It reads three structurally incompatible network traffic datasets, cleans each independently, resolves label conflicts through a formal taxonomy, and produces unified Parquet output files with 11 mandatory metadata fields. Nothing in Stage 1 is fitted on any subset - it operates on raw data only. Its only job is to produce clean, labelled, schema-valid records.

2.1  The Three Datasets
Each dataset was collected under different conditions, using different capture tools, and labelled using different taxonomies. This heterogeneity is precisely why multi-source training is valuable - a model trained on all three datasets must learn features that generalise across different network environments and labelling philosophies.

Why these three datasets?
The three datasets together provide temporal diversity (1999 ? 2015 ? 2017), capture-method diversity (Tcpdump ? PCAP with argus ? CICFlowMeter), and attack-category diversity. No single dataset covers all modern attack families. NSL-KDD contributes well-labelled classic attacks; UNSW-NB15 contributes modern reconnaissance and exploit patterns; CICIDS-2017 contributes the largest volume of DDoS, web, and infiltration attacks. A model trained only on one would fail badly on the others.

The 99th-percentile replacement values for Flow Bytes/s (? 12,333,333) and Flow Packets/s (? 2,000,000) were computed on the full CICIDS-2017 corpus before the Stage 2 train/test split.

Why apply Quality Gates at ingestion rather than preprocessing?
Separating quality gating from preprocessing ensures that every downstream stage receives data with documented provenance. If a Stage 3 model produces anomalous results, the quality_flags metadata column on every record allows engineers to filter out QG-03 or QG-05 records and verify whether those flags explain the anomaly. Embedding quality decisions inside the preprocessing notebook would hide them from downstream debugging.

A five-class master taxonomy was designed to capture operationally meaningful threat categories that exist in all three datasets:

Why a 5-class taxonomy specifically?
Five classes balances granularity with trainability. More classes (e.g. 15 CICIDS-specific attack names) would create extreme class imbalance - Heartbleed has only 11 records across all datasets. Fewer classes (e.g. binary: normal vs attack) would lose the operational value of distinguishing DoS from intrusion from malware.
The five chosen categories (NORMALL, DoSD, PROBE, EXPLOIT, MALWARE) map directly to the MITRE ATT&CK framework's initial access, discovery, lateral movement, and impact phases - making the model's output actionable for SOC analysts.

3. Stage 2 - Classical Preprocessing Pipeline

Stage 2 transforms the three Stage 1 Parquet files into a single, clean, normalised, class-balanced feature matrix ready for deep learning. It is the most complex stage in the pipeline because it must solve six distinct problems simultaneously: heterogeneous feature spaces, categorical encoding, cross-dataset absent features, instrument artefacts, true missing values, and class imbalance - all without letting any test-set information leak into any fitted parameter.

Confirmed runtime result: 2,367,588 training samples × 167 features and 573,807 test samples × 167 features. All 8 output validation assertions passed.

3.1  Pipeline Execution Order and the Leakage-Prevention Firewall
The single most important design constraint in Stage 2 is that the train/test split must happen before any fitting operation. Every encoder, scaler, and imputer must be fitted exclusively on training data. The pipeline enforces this through strict sequential ordering.

The Leakage-Prevention Firewall
Step 3 (Train/Test Split) is the firewall. Nothing before it fits anything. Everything from Step 4 onward fits only on training data.
If any encoding, scaling, or imputation were applied before the split, the model's test-set metrics would appear better than they truly are - the model would have 'seen' test data during training.
This is the most common mistake in ML preprocessing pipelines and the one most likely to be caught by paper reviewers.

3.2  Step-by-Step Deep Dive
Step 1 - Load Stage 1 Outputs and Schema Validation
Each Parquet file is loaded and immediately validated. The validation checks two things: the schema_version field is exactly '1.0', and all 11 mandatory metadata columns are present. This gate prevents a common failure mode where a re-run of Stage 1 with different parameters silently produces a different schema that Stage 2 then processes incorrectly.

Confirmed output: NSL-KDD 147,907 records (36 features), UNSW-NB15 198,802 records (42 features), CICIDS-2017 2,522,326 records (77 features). Total: 2,869,035.

Step 2 - Feature Alignment: Padding to the 154-Column Union
The most fundamental structural challenge of multi-source training: the three datasets have completely different feature sets. A model cannot receive different numbers of inputs for different records. The solution is to define a unified feature space as the union of all features across all datasets, then fill 'absent' feature slots with sentinel values.

Sentinel Policy (Design Decision D7-1)
Feature TypeSentinel UsedPost-Scale ValueWhy This ValueNumeric-1.00.0Network metrics are always = 0. -1.0 is unambiguously 'absent', not a valid feature value.Categorical (string)'__ABSENT__'0.0 (all-zeros OHE or 0 frequency)A string sentinel cannot be confused with real category values. Converts to all-zero vector after encoding.
Why not delete absent features and train three separate models? Because the research goal is a single unified quantum model that generalises across datasets. Separate models would not produce a shared 8D quantum-compatible representation, and ensemble-ing three separate quantum circuits is computationally infeasible. The sentinel approach allows one model to learn 'this feature is absent for this dataset' as a structured input, not a missing value.

Confirmed output: Unified DataFrame: 2,869,035 records × 165 columns (154 features + 11 metadata). NSL-KDD sentinel rate 76.6%, UNSW-NB15 72.7%, CICIDS-2017 50.0%.

Step 3 - Dual-Stratified Train / Test Split
The split must preserve two dimensions of balance simultaneously: class distribution (master_label) and dataset provenance (source_dataset). A single-dimension stratification on label alone would allow, for example, the test set to draw disproportionately from CICIDS-2017's NORMALL records while underrepresenting NSL-KDD's NORMALL records - producing a test set that doesn't reflect the true multi-source distribution.

Compound Stratification Key
The stratification key is the concatenation: master_label + '__' + source_dataset. This creates 14 strata (MALWARE does not exist in NSL-KDD, reducing the theoretical 15 to 14). StratifiedShuffleSplit guarantees each stratum appears in both train and test at exactly the 80/20 ratio.

Why StratifiedShuffleSplit rather than train_test_split with stratify?
sklearn's plain train_test_split with stratify= only handles a single stratification column. It cannot stratify on a compound key with 14 values without manual construction. StratifiedShuffleSplit natively handles arbitrary compound keys and provides a cleaner API for reproducible splitting (fixed random_state=42 across all runs).

Step 4 - Categorical Encoding
Five categorical columns exist across the three datasets. Each must be converted to numeric values before any mathematical operation. Two encoding strategies are applied based on column cardinality, with a threshold of 10 unique non-sentinel values.

The FrequencyEncoder: Design and Why Not Target Encoding
Frequency encoding maps each category value to its proportion in the training set. For example, if 'http' appears in 42% of NSL-KDD service values, it maps to 0.42. Unseen test values map to 0.0. The __ABSENT__ sentinel maps to -1.0 (the sentinel value), which is later zeroed by the sentinel mask.

Target encoding (replacing a category with its mean label value) was considered and rejected. With NORMALL at 79% of training data, most categories' mean label would be close to 0 regardless of their actual threat relevance. Target encoding also leaks label information into the feature, which is statistically valid for tree models but problematic for neural networks where gradient updates propagate label information back through encoded features.

The OHE Sentinel Handling Fix (CF-S2-02)
A non-obvious implementation challenge: fitting OHE only on non-sentinel rows fails with 'ValueError: 0 samples' when every training row has at least one sentinel column (because different rows have __ABSENT__ in different columns). The fix uses a guaranteed out-of-vocabulary placeholder string '__OHE_ABSENT__', fits OHE on all rows (replacing __ABSENT__ with the placeholder), and relies on handle_unknown='ignore' to produce all-zeros for the placeholder. Rows with __ABSENT__ are then forced to all-zeros as a safety verification step.

Step 5 - Sentinel Mask Creation
The sentinel mask is a boolean array of shape (N × 167). A True at position [i, j] means: 'feature j was not measured for record i because record i comes from a dataset that does not include feature j as part of its capture methodology.' This is fundamentally different from a missing value - it is a structured, expected absence.

Must be created BEFORE any scaling (D7-1)
The SentinelAwareMinMaxScaler (Step 9) uses this mask to exclude sentinel positions from its per-column min/max computation.
If the mask were created after scaling, the scaler would have already used the -1.0 sentinel values to compute column statistics, compressing all real feature values into a sub-range of [0,1].
The mask is built from metadata (source_dataset column) and the feature lineage dictionary, not from feature values - so there is no fitting and no leakage.
Construction Algorithm
For each dataset (NSL-KDD, UNSW-NB15, CICIDS-2017): determine which post-encoding column names have a pre-encoding parent that was not in that dataset's original feature set. For all records from that dataset, mark those column positions as True in the mask. OHE-derived columns inherit the sentinel status of their parent column via the feature_lineage dictionary.

Confirmed runtime: Training mask: 2,295,228 × 167 = 383,202,876 positions total, 215,042,933 True (56.1% sentinel). NSL-KDD rows: 15,145,728 sentinel positions, UNSW-NB15: 18,289,715, CICIDS-2017: 181,607,490.

Verification note: The notebook checks that sentinel_positions match correct_sentinel_val. For NSL-KDD: 15,145,728 sentinel positions but only 13,844,142 have the value -1.0. The discrepancy (˜1.3M positions) comes from OHE-derived columns: OHE produces 0.0 for absent rows (all-zeros), not -1.0 - so those positions are correctly absent but their value is 0.0 rather than -1.0. The sentinel mask correctly flags them regardless of value.

Step 6 - CICFlowMeter Integer-Overflow Clipping
CICIDS-2017 was generated using CICFlowMeter, which has a documented signed 32-bit integer overflow bug. When computed flow statistics exceed INT32_MAX (2,147,483,647), the value wraps to a large negative number. Stage 1 flagged 50.7% of CICIDS-2017 records with bounds_violation=True because of this - but correctly did not modify the values, deferring correction to Stage 2.

Why clip rather than exclude?
Excluding all 50.7%-flagged records would remove 1.28 million CICIDS-2017 records - nearly half the dataset.
The overflow is an instrumentation artefact in a small number of computed columns. The underlying network flows are real and valid.
Clipping to domain-valid bounds (e.g. 0 = header_length = INT32_MAX, -1 = TCP_window = 65535) recovers the records without introducing fabricated values.
The clip bounds are derived from TCP/IP protocol specifications - they are not learned from data, so there is no leakage from applying the same bounds to train and test.
Confirmed runtime: 703 values corrected in train, 176 in test. Affected columns: Fwd Header Length (24 train), Bwd Header Length (15 train), Fwd Header Length.1 (24 train), min_seg_size_forward (24 train), Flow Duration (87 train), Flow Bytes/s (355 train), Flow Packets/s (87 train), Flow IAT Mean (87 train). Init_Win_bytes columns had no overflow values - already within [-1, 65535].

Step 7 - Median Imputation for True Missing Values
After alignment and sentinel padding, the only genuine NaN values are in the Flow Bytes/s column for CICIDS-2017 records where CICFlowMeter failed to compute a byte rate (typically zero-duration flows that produce division by zero). These records were flagged has_missing=True in Stage 1.

Stage 1 Forward Requirement #5 specifies that SMOTE must not be applied to has_missing records without prior imputation - SMOTE's nearest-neighbour calculation is undefined over NaN values.

Step 9 - Sentinel-Aware MinMax Scaling
All feature values must land in [0.0, 1.0] before the VAE. The VAE bottleneck output is then scaled to [0, p] for quantum angle encoding - so the model is sensitive to absolute input scale. Standard MinMaxScaler cannot be used here because it would incorporate the -1.0 sentinel values into its per-column min/max statistics.

Why Standard MinMaxScaler Fails
Standard MinMaxScaler computes: scaled = (x - min) / (max - min). If a column has sentinel value -1.0 and real values in [0, 1000], the scaler computes min = -1.0. After scaling, real values map to [0.001, 1.001] - but when we then clip to [0, 1], values exactly at the real minimum (0) map to 0.001 and are compressed. More importantly, the sentinel value maps to 0.0 - which is indistinguishable from a real zero. Tracking the sentinel mask separately solves the ambiguity, but the scaler statistics are still wrong.

SentinelAwareMinMaxScaler: Architecture
class SentinelAwareMinMaxScaler:
    def fit(self, X, sentinel_mask):
        for j in range(n_features):              # iterate over columns
            for chunk in row_chunks(X[:, j]):     # 50K rows at a time (~400 KB)
                valid = chunk[~sentinel_mask_chunk]  # exclude sentinel rows
                finite = valid[np.isfinite(valid)]   # exclude NaN/Inf
                update col_min[j], col_max[j]         # running min/max
 
    def transform(self, X, sentinel_mask, out_path=None):
        # out_path ? write to disk-backed F-order memmap (0 GB RAM for output)
        for j in range(n_features):
            for chunk in row_chunks(X[:, j]):
                scaled = (chunk - col_min[j]) / scale[j]
                np.clip(scaled, 0.0, 1.0, out=scaled)    # in-place, no copy
                scaled[sentinel_mask_chunk] = 0.0         # zero sentinel positions
                X_scaled[chunk_rows, j] = scaled
Memory strategy: The full 2,295,228 × 167 float32 matrix weighs 1.43 GB. Storing it in RAM would exhaust available memory. The scaler writes output to a disk-backed F-order memory-mapped file, processing 50,000 rows at a time (~400 KB peak intermediate). Column j of the memmap is stored contiguously on disk (F-order), so reading X_scaled[:, j] for verification is a single sequential seek.

Post-scaling invariants (all verified): (1) All non-sentinel positions in [0.0, 1.0]. (2) All sentinel positions exactly 0.0. (3) No NaN or Inf. The 0.0 value for both 'real feature = 0' and 'feature absent' is disambiguated by the sentinel mask - both consumers (VAE and ensemble) receive the mask alongside the feature array.

Confirmed runtime: SentinelAwareMinMaxScaler fitted on 167 features. All 3 verification checks passed column-by-column.

SMOTE Memory Challenge and Solution
Standard SMOTE applied to the full 2,295,228 × 167 training matrix would attempt to load the entire matrix into RAM (1.43 GB) plus the resampled output - exceeding available memory. The solution extracts only the minority-class rows and a majority-class anchor (needed for imbalanced-learn's internal validation) into a compact 152,641 × 167 subset (102 MB), runs SMOTE on that subset, extracts only the synthetic rows, and streams the full augmented dataset to a new disk-backed memmap.

Key insight: SMOTE's KNN computation searches only within each minority class - majority rows are never involved in nearest-neighbour calculations. Including a majority anchor is only required to satisfy imbalanced-learn's internal validation check. The synthetic samples generated from a subset are bit-for-bit identical to those that would be generated from the full training set.

6. Near-Zero Variance Filtering - Missing Step

Stage 1 Residual Item #2 explicitly required Stage 2 to drop 12 near-constant CICIDS-2017 features via variance filtering before scaling.

6.1  What Near-Zero Variance Features Are
A near-zero variance feature is a column where the vast majority of values are identical across all records. A binary TCP flag counter that is 1 for 0.001% of records and 0 for the other 99.999% has variance ˜ 0 and carries almost no information. These 12 columns are artefacts of the CICFlowMeter capture methodology - certain TCP control flags were almost never triggered in the CICIDS-2017 lab environment, leaving those counters at zero for every practical sample. Stage 1 retained them faithfully (its job is raw preservation). Stage 2 is where uninformative features are removed before modelling.

6.3  Why This Is Critical for the 8D VAE Bottleneck
The VAE must compress 167 features into exactly 8 dimensions. That bottleneck feeds directly into the VQC as qubit rotation angles. Three cascading problems arise from feeding near-constant columns into this compression:

1. The VAE reconstruction loss is dominated by the easy-to-reconstruct near-constant columns. Outputting near-zero for these 12 columns achieves low loss without learning anything about attack patterns.
2. Backpropagation through 167 features means the 12 near-constant columns contribute gradient noise competing with the 155 informative features for the same 8 bottleneck neurons.
3. A qubit rotation angle that encodes 'this flow had zero URG flags' is a wasted qubit - URG is zero for essentially every flow. That qubit cannot simultaneously encode a discriminative attack signal.

Quantified impact: 12 near-zero variance columns out of 77 CICIDS-2017 features = 15.6% of the CICIDS real feature space wasted. Since CICIDS-2017 records represent 87.9% of the corpus, these 12 columns appear in the dominant majority of training samples and influence the majority of reconstruction-loss gradient updates.

6.4  Why Standard VarianceThreshold Cannot Be Used Directly
sklearn's VarianceThreshold computes variance over all values in a column, including the -1.0 sentinel values assigned to NSL-KDD and UNSW-NB15 records (which don't have these CICIDS-only columns). A column with value 0 for 2M CICIDS records and -1.0 for 350K NSL+UNSW records appears to have substantial variance - driven entirely by the sentinel, not by real information. The standard tool would therefore fail to drop these columns even though their real-data variance is near-zero.

The correct approach computes per-column variance only over non-sentinel rows using the sentinel mask (sentinel_mask[:, j] == False). For the 12 CICIDS-only columns, this means computing variance only over the ~2M CICIDS-2017 training records, which gives an honest near-zero result.
